## Technical Challenges & Priorities

### P0: The Monolith Problem (Biggest Challenge)

**What it actually is:**
> "It's not really the code base that's the problem — it's everything that comes with it."

**Real Issues:**
- **Monolithic team structure** (until recent reorg)
- **Monolithic deployment process** — Once per month cycles
- **Monolithic test suite** — 2-3 weeks to run full regression
- **Monolithic infrastructure footprint**

**Current Pain:**
- Deploy entire system once a month
- Spend 2-3 weeks every cycle on testing
- Fix broken UI tests as one large package
- Incredibly slow delivery

**Goal: Autonomous Teams**
- 12 teams able to deploy independently
- Own their own destiny, pipelines, test suites
- Even if still on "mini-monolith" initially
- Massive win if teams can deploy on demand

**Philosophy:**
> "When people talk about the monolith, most think about the code base. But it's the organization, the processes, and the deployment footprint that's actually more important."

---

### P0: Data Layer (Stability #1 Threat)

**Current State:**
- Most data in **MongoDB**
- "15 years ago Mongo was given to us as a hammer, since then everything looks like a nail"
- Used for almost everything, whether appropriate or not

**Problems:**
1. **Wrong tool for many jobs** — Read/write patterns don't match MongoDB strengths
2. **Biggest threat to stability** — Most medium/major incidents trace to data layer
3. **Noisy neighbor problems** — Everything on one cluster
4. **Cost implications** — Over-reliance driving up Atlas costs
5. **Toppling over** — MongoDB failures cascade

**Strategy:**
- Moving from self-hosted to managed services
- Breaking some clusters apart (1 cluster → 3)
- **First pass:** NOT touching data layer during app layer decoupling
- Teams will initially share databases, take ownership later once operating independently
- Data layer changes less frequently than app layer, so less of a blocker for day-to-day work

**Var's Concerns:**
- Over-reliance on MongoDB hypothesis confirmed during diligence
- Build spike likely related to Atlas migration
- Kickdrum already asked to make recommendations on right database tech for right problems
- **Critical decision:** Granularity of databases (200 microservices ≠ 200 databases)

---

### P1: Event Bus (Homegrown, Lives in Mongo)

**Current State:**
- Producers write to Mongo collection
- Consumers do `SELECT * WHERE matches_me` polling
- Very homegrown, not proper eventing

**Needs:** Replace with proper messaging system (Kafka, SQS, etc.)

---

### P1: Identity Provider (Homegrown SSO)

**Current State:**

- Started ~15 years ago (no good alternatives then)
- Used by other applications

**Should be:** Off-the-shelf (Auth0, Okta, etc.)

---

### P1: Observability (Gaps & Pain Points)

**In Progress:**

**Future State:**

---

### P0: Disaster Recovery (Top of Mind)

**Challenge:** More work than anyone wants because much infrastructure is NOT in managed services

**Current Approach:**
- **NOT re-architecting** at this stage
- Ensure current infrastructure covered by IaC
- Good runbooks
- Make current solution disaster-recoverable

**Targets:**
- RPO: Meeting easily (~15 minutes)
- RTO: 4 hours

**Failover Model:**
- All of Exchange fails over together, or nothing
- Exchange must be live in same region
- **Not ideal state** — just first phase

**Future State (Aspirational):**
- System runs **multi-region every day**
- Many AWS managed services provide this for free
- Self-healing network effect (50 nodes fail → recover just those, not all 200)
- Requires decoupling via async queues/topics

**Var's Recommendations:**
- Push Kickdrum to understand current architecture AND future thinking
- Get external validation on approach
- Two benefits:
  1. Opportunity to collapse timeline, pull things left
  2. Endorsement that complexity requires time → creates peace for product team

### Blocking I/O Concerns

**Var's Warning:**
> "If you're writing with blocking I/O, databases will become a bottleneck. 100 services writing in parallel to one choke point → lots of waiting."

**Considerations:**
- Need non-blocking I/O architecture

- At GHX's scale, this becomes a **computer science problem**, not just CRUD
