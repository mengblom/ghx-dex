## Meeting Summary

### **Main Incident: Audit MongoDB Database Issues (Sev 1-2)**

**Root Cause:**
- Upgraded MongoDB from version 4.2 to 4.4 in January for Atlas migration prep
- Upgrade introduced deep engine changes to WiredTiger History Store
- Caused intermittent WiredTiger ticket exhaustion and dirty cache issues
- **Resolution:** Rolled back to MongoDB 4.2.25 on Thursday - system stabilized

**Key Discussion Points:**
- Problem was intermittent for 3-4 weeks before becoming critical
- Lower environments (dev, stage, ATM) didn't show the same issues due to different load patterns
- MongoDB vendor had limited ability to support the 10-year-old database
- 15-20 major customers impacted in final week, prompting rollback decision

**Path Forward:**
- Development team assessing options for creating "headroom" before next upgrade attempt
- Meeting with MongoDB scheduled to discuss recommendations
- Options range from 2-6 months of work
- Focus on work already planned post-migration (pulling forward roadmap items)
- Timeline: Tentative plan by early next week, decisions by end of next week
- **Cannot upgrade to 4.4 again** - may need to jump directly to version 6 or 8
- Elasticsearch migration postponed due to resource constraints

**Major Concerns:**
- Current testing can't guarantee production stability
- Atlas may have LESS capacity per node than current on-prem setup
- Need significant buffer, not just "enough to squeak by"
- Must address architectural issues: 188M ActivityFlow documents/day written then deleted 3 days later
- Bad dependencies: SFTP endpoints and core processing shouldn't fail when Audit DB is unresponsive

---

### **Secondary Incident: WFE Worker Configuration (Sev 3)**

**Issue:**
- WFE (Web Front End) configured for 2,048 workers but only running 400 (Apache default)
- Contributed to 504 errors during database incident
- Same issue previously affected AUI, was fixed by security team in March 2025

**Resolution Plan:**
- Infra team (Graham/Kevin) taking ownership from security team
- Will rebuild with latest AMIs and proper worker configuration

---

### **Third Incident: EU DocStore/Oracle Database Outage (Sev 3)**

**Issue:**
- EU-only outage Saturday afternoon through Monday morning
- Oracle database ran out of archive logs, became unresponsive
- Affected J&J and some GFX traffic using DocStore service
- Delay in response due to insufficient monitoring/alerting

**Root Cause:**
- Oracle version 11 (20+ years old, unsupported since August)
- No Oracle support contract - "best effort only"
- No proper monitoring (would require Oracle Enterprise Manager license)
- Only Avinash has access; no Oracle DBAs on staff
- Matthew Turner only dev who knows service (but knows "nothing about it")

**Resolution Plan:**
- Matthew and René to assess current usage by March 20
- Ali to investigate monitoring options within 1 week
- Service on "kill list" - Marten and Daniel planning to decommission as part of legacy cleanup
- DocStore has "daily outages" requiring restarts

---

### **Organizational Updates:**
- **Gaurav Saini** introduced as new ITSM Manager
- Building 24/7 ITSM team with 3 people from current Exchange NOC
- Moving toward L1 support function
- **Munib** continuing as Senior SRE Manager

---

### **Key Takeaways:**
1. **Technical Debt Crisis**: Multiple unsupported legacy systems (MongoDB 4.2, Oracle 11) causing production incidents
2. **Testing Limitations**: Cannot replicate production load patterns in lower environments
3. **Urgent Need**: Must create architectural improvements AND upgrade path
4. **Customer Impact Philosophy**: Even 3-4% of customers = major incident if they're large customers
5. **Resource Constraints**: Same teams working Audit DB issues also handle Elasticsearch migration
